# Symbolic grading

## Floating point arithmetic

**Floating point arithmetic** is
how computers usually perform arithmetic.
It works on a **finite subset** of rational numbers,
that are not uniformally distributed.
In particular, $0$ is the value around which
floats are more concentrated,
as evidenced by the following calculations:

:::example

<div class="grid grid-cols-2 gap-8">
  <div>
    <Code lang="python" value="10 ** (-18) / 2" run />
  </div>
  <div>
    <Code lang="python" value="1 + 10 ** (-16)" run />
  </div>
</div>
:::

The use of base $2$,
combined with the use of finitely many significant digits,
mean that numbers such as

$$
\frac 1 {10} = (0.0\overline{0011})_2
$$

will not be exact in the computer's memory.
This will lead to error propagation:

<Code lang="python" value="0.1 * 0.1" run />

As we also need to be able to manipulate symbols,
floating point arithmetic is unsuitable
for any meaningful pedagogical application.

## Symbolic grading

Symbolic grading consits in using a
[Computer Algebra System](https://en.wikipedia.org/wiki/Computer_algebra_system)
(such as [GeoGebra](https://geogebra.org) or [SymPy](https://www.sympy.org/))
for determining
if the student's attempt is correct.
CAS work by representing expressions
as [Abstract Syntax Trees](https://en.wikipedia.org/wiki/Abstract_syntax_tree).

<div class="flex items-center justify-around">
  <MathField value={value()} oninput={(e) => setValue(e.target.value)} />
  <MathJson value={value()} />
</div>

Unfortunately,
[Computer Algebra Systems](https://en.wikipedia.org/wiki/Computer_algebra_system)
often suffer from at least one of the following drawbacks.

- They are not **designed for the Web**:
  CAS are very often not written in JavaScript,
  The interface is therefore slower for the student,
  as all work must be sent over the wire to a server.
  This also raises issues scaling issues for the developer
  as they need to manage the network boundary.

  A notable exception is MathLive's [Compute Engine](https://mathlive.io/compute-engine/).
  However, it is unable to **factor an expression**.

- They perform **automatic simplifications**,
  without any possibility for opting out.
  This means we lose information on the student's attempt
  as soon as the CAS parses it.

  Unfortunately,
  there are no exceptions that I'm aware of.

## Current solution

We use both [Compute Engine](https://mathlive.io/compute-engine/)
and [SymPy](https://sympy.org).
The pipeline is as follows:

1. [Compute Engine](https://mathlive.io/compute-engine/)
   parses the student's input into a tree on the **client**.
   Its [MathJSON](https://mathlive.io/math-json/) syntax
   for the AST is simple, extensible,
   and suited for network communications.

   :::example
   MathJSON representation of <MathField value={value()} oninput={(e) => setValue(e.target.value)} />

   <div class="flex items-center justify-around gap-4">
     <MathJson value={value()} />
     <Code class="grow" lang="json" value={JSON.stringify(expr(value()).json, null, 1)} />
   </div>
   :::

2. `@learning/core` is a thin TypeScript library
   that is part of the _learning_ project.
   It will decide which CAS to use depending on the situation,
   by favouring **Compute Engine** as much as possible,
   and being **lazy** for any work that requires SymPy.

   :::example

   To get a sense of what it does, we give the following examples
   - Substitutions are handled well by both Compute Engine and SymPy,
     so `@learning/core` delegates to Compute Engine.
     There is no client/server request.

   - Factoring expressions can only be done by SymPy.
     `@learning/core` will add `Factor` as the root node,
     and do nothing else.

     <div class="flex items-center justify-around">
       <MathJson value={value()} />
       <MathJson value={['Factor', expr(value()).json]} />
     </div>

     Again, there is no client/server request,
     but Compute Engine is now unable to understand the expression,
     as it does not know the `Factor` instruction.

   - Checking equality and getting a $\LaTeX$ can only be reliably done on SymPy.
     We send a request with the operation to perform, the MathJSON of the expressions,
     and wait for the response.

   :::

## API

:::example

```typeScript
import { expr } from '@learning/core'

const attempt = expr('(x - 2)(x - 3)')

await attempt.isEqual('x^2 - 5x + 6') // true
await attempt.isFactored() // true
await attempt.diff().latex() // 2x - 5
await attempt.expand().latex() // x^2 - 5x + 6
```

:::
